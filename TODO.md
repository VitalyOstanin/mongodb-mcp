## Compose-стенд для интеграционных тестов

Цель: воспроизводимый локальный и CI-friendly стенд с реально запущенной MongoDB, чтобы прогон интеграционных тестов перестал зависеть от внешней БД.

Решения, которые надо принять до реализации:

- Базовый рантайм -- `podman-compose` (предпочтительный, согласно глобальным правилам), с fallback-инструкцией для `docker compose`. Файл -- `compose.yaml` (новый формат, без `version:`).
- Версия сервера -- зафиксировать конкретный тег `mongo:8.0.6` (или ближайший актуальный patch на момент мерджа). Не использовать `mongo:latest` -- это даст плавающие падения CI при выходе нового образа.
- Том для данных -- `tmpfs` или анонимный volume; БД пересоздаётся между прогонами, чтобы не тащить грязное состояние между PR.
- Аутентификация -- включить с тестовыми credentials (`MONGO_INITDB_ROOT_USERNAME` / `MONGO_INITDB_ROOT_PASSWORD`), чтобы тесты ходили по нормальному пути авторизации, а не по anonymous-режиму.
- Healthcheck в `compose.yaml` через `mongosh --eval 'db.runCommand({ ping: 1 })'`, чтобы тесты не стартовали до готовности сервера.
- Сетевой контракт -- сервис слушает `127.0.0.1:27017` (без публикации в `0.0.0.0`), чтобы случайно не открыть БД наружу.

Открытые пункты:

- Решить, нужна ли отдельная роль с минимальными правами для тестов (предположение: да, иначе мы всегда тестируем root-сценарий и пропустим ошибки авторизации).
- Решить, шарим ли мы один стенд с задачей CI #36 (integration tests в GitHub Actions), либо в CI используем `services:` блок workflow, а в локальной разработке -- `compose.yaml`.

## README: Security considerations

Цель: добавить в README (и `README-ru.md`) раздел "Security considerations" с явными рекомендациями для пользователей.

Содержание раздела (минимум):

- Обязательность TLS для удалённых кластеров: `mongodb+srv://` либо явный `tls=true` в обычном connection string.
- Использование MongoDB-роли с минимальными привилегиями: read-only на нужных БД, запрещены `eval` / scripting, никаких admin-привилегий, если они не используются.
- Хранение connection string в `.env` (через `MONGODB_MCP_CONNECTION_STRING`), а не в `args` команды запуска MCP-клиента -- иначе пароль попадёт в логи запуска MCP host.
- Запрет на серверный JS: `$where`, `$function`, `$accumulator` уже заблокированы в коде; дополнительно желательно запускать сервер MongoDB с `--javascriptEnabled=false`.
- Ограничение `MONGODB_MCP_EXPORT_DIR` на отдельный каталог, к которому нет доступа у других пользователей системы (chmod 700) -- write-инструменты сбрасывают данные туда.

Источник: security-review в [docs/reviews/2026-05-05-2104-review.md](docs/reviews/2026-05-05-2104-review.md), пункты про TLS и `[info] Connection string без TLS`.

## README: Concurrency considerations

Цель: задокументировать ограничения по параллельным операциям, чтобы LLM-клиент и пользователь не ждали транзакционных гарантий, которых нет.

Содержание раздела (минимум):

- Tools -- одиночные операции; они не объединены в транзакции и не разделяют `ClientSession`. Multi-step сценарии надо проектировать атомарно: класть проверочные условия в `filter` команды `update`/`delete`, а не делать "сначала `find`, потом `update`".
- Singleton MongoDB-клиента сериализует только сами `connect`/`disconnect` через мьютекс. Параллельные обычные операции (find/aggregate/insert и т.п.) идут одновременно через один MongoClient -- никакого упорядочивания их относительно друг друга MCP-сервер не делает.
- `getConnectionInfo()` -- snapshot, а не атомарное состояние; для критичных решений полагаться на результат самой операции, а не на предварительный `isConnected`.
- Запись в файлы (`saveToFile`) использует `wx`-флаг и не перезапишет существующий файл -- параллельные tool-вызовы получат `EEXIST` и должны выбрать другой `filePath`.

Источник: database-race в [docs/reviews/2026-05-05-2104-review.md](docs/reviews/2026-05-05-2104-review.md), пункты про Singleton MongoDBClient и Concurrency considerations.
